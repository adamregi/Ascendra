# AI Rules — Ascendra

> **Purpose**: Explicit prohibitions and constraints for all AI coding agents working on Ascendra.
> **Severity**: Violations of these rules indicate a failed implementation.

---

## Absolute Prohibitions

### 1. Never Compute Backend-Owned Analytics in Flutter

```
❌ PROHIBITED:
  final totalMeetings = meetings.length;
  final avgAttendance = meetings.map((m) => m.attendees).reduce((a, b) => a + b) / meetings.length;

✅ REQUIRED:
  final dashboard = await repo.getDashboardStats();  // RPC call
  // dashboard.totalMeetings, dashboard.avgAttendance — pre-computed by backend
```

All KPIs, scores, percentages, trends, and aggregations MUST come from RPCs or materialized views.

### 2. Never Bypass Repositories

```
❌ PROHIBITED (in widget or provider):
  final data = await Supabase.instance.client.from('profiles').select();

✅ REQUIRED:
  final data = await ref.read(memberRepositoryProvider).getMemberDirectory();
```

All data access flows through: Provider → Repository → Supabase/Dio.

### 3. Never Access Supabase Directly from Widgets

```
❌ PROHIBITED:
  class MyWidget extends StatelessWidget {
    Widget build(context) {
      final client = Supabase.instance.client;  // NEVER
    }
  }

✅ REQUIRED:
  class MyWidget extends ConsumerWidget {
    Widget build(context, ref) {
      final data = ref.watch(myProvider);  // Provider handles data access
    }
  }
```

Widgets are presentation-only. They receive typed view models and render UI.

### 4. Never Ignore the Skills/ Folder

Before implementing any feature, the AI MUST:
1. Identify relevant skill categories
2. Search the Skills/ directory
3. Read matching SKILL.md files
4. Apply skill guidance alongside project architecture

The user should NEVER need to remind the AI to check skills.

### 5. Never Invent UI Layouts When Reference Assets Exist

```
❌ PROHIBITED:
  // Building a meetings page without checking reference
  class MeetingsPage extends StatelessWidget { ... }

✅ REQUIRED:
  // First: inspect assets/reference/meetings_command_center/screen.png
  // Then: inspect assets/reference/meetings_command_center/code.html
  // Then: implement matching the reference
```

If `assets/reference/<screen>/` exists, the implementation MUST match it.

### 6. Never Duplicate Reusable Components

Before creating a new widget, check:
1. `lib/shared/widgets/` — 16 shared components
2. `lib/core/widgets/` — core components
3. The feature's own `widgets/` directory

If an existing component handles the use case (even with minor adaptation), reuse it.

### 7. Never Modify Migrations Without Following the Migration Guide

```
❌ PROHIBITED:
  // Editing an existing migration file
  // Modifying 028_helper_functions.sql directly

✅ REQUIRED:
  // Create a NEW migration file with the next sequence number
  // e.g., 061_fix_member_profile_rpc.sql
```

Migrations are immutable once applied. Always create new sequential files.

### 8. Never Use Raw Strings for JSON Parsing

```
❌ PROHIBITED:
  final name = json['first_name'] as String;
  final age = json['age'] as int;

✅ REQUIRED:
  final model = MyModel.fromJson(json);  // Generated by json_serializable
```

All JSON parsing uses Freezed + json_serializable generated code.

### 9. Never Create Non-Code-Generated Providers

```
❌ PROHIBITED:
  final myProvider = Provider<MyService>((ref) {
    return MyService();
  });

✅ REQUIRED:
  @riverpod
  MyService myService(Ref ref) {
    return MyService();
  }
```

All providers use `@riverpod` annotation with code generation.

### 10. Never Skip Error Handling in Repositories

```
❌ PROHIBITED:
  Future<Model> getData() async {
    final response = await _client.rpc('my_rpc');
    return Model.fromJson(response);
  }

✅ REQUIRED:
  Future<Model> getData() async {
    try {
      final response = await _client.rpc('my_rpc');
      return Model.fromJson(response as Map<String, dynamic>);
    } catch (e, stack) {
      handleException(e, stack);  // BaseRepository method
    }
  }
```

### 11. Never Hardcode Colors, Sizes, or Fonts

```
❌ PROHIBITED:
  color: Color(0xFF181e1a)
  fontSize: 16
  fontFamily: 'Inter'

✅ REQUIRED:
  color: AppColors.primary
  style: AppTypography.body1
  padding: EdgeInsets.all(AppSpacing.md)
  borderRadius: BorderRadius.circular(AppRadius.lg)
```

Always use design system tokens from `core/constants/`.

### 12. Never Create Mutable Data Models

```
❌ PROHIBITED:
  class UserModel {
    String name;
    int age;
  }

✅ REQUIRED:
  @freezed
  class UserModel with _$UserModel {
    const factory UserModel({
      required String name,
      required int age,
    }) = _UserModel;

    factory UserModel.fromJson(Map<String, dynamic> json) =>
        _$UserModelFromJson(json);
  }
```

All data models are immutable Freezed classes.

When making implementation decisions:

1. **Explicit user instructions** — always highest priority
2. **AGENTS.md rules** — project operating manual
3. **This file (AI_RULES.md)** — explicit prohibitions
4. **Relevant Skills/** documents — project-specific guidance
5. **Existing project patterns** — consistency with current code
6. **Official framework documentation** — Flutter, Riverpod, Supabase
7. **General software engineering best practices** — lowest priority

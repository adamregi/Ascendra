# Database Guide — Ascendra

> **Purpose**: Conventions and best practices for interacting with and designing the Ascendra PostgreSQL database.

---

## 1. Schema Conventions

### Naming
- Tables: `snake_case`, plural (e.g., `tasks`, `profiles`).
- Columns: `snake_case` (e.g., `first_name`, `created_at`).
- Primary Keys: `id` (always `UUID`, generated by `uuid_generate_v4()`).
- Foreign Keys: `<table_singular>_id` (e.g., `company_id`, `assignee_id`).

### Standard Columns
Every table MUST have:
- `id uuid primary key default uuid_generate_v4()`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

If the table holds tenant data, it MUST have:
- `company_id uuid not null references companies(id) on delete cascade`

## 2. Row-Level Security (RLS)

RLS is **mandatory** for every table. 

### Enabling RLS
```sql
alter table public.tasks enable row level security;
```

### The Tenant Isolation Policy
This policy must exist on every table that belongs to a specific company.

```sql
create policy "Company isolation for tasks"
  on public.tasks
  for all -- Or select, insert, update, delete
  using (company_id = public.get_user_company_id());
```

## 3. Data Types

- **Dates/Times**: Always use `TIMESTAMPTZ` (Timestamp with time zone). Never use `TIMESTAMP`.
- **Strings**: Use `TEXT`. Add `CHECK` constraints if the string should be an enum.
- **Hierarchies**: Use `ltree`.
- **JSON**: Use `JSONB` for unstructured data or settings.

### Enums via CHECK Constraints
We use `TEXT` with `CHECK` constraints instead of PostgreSQL `ENUM` types because modifying constraints is much easier in production than altering enum types.

```sql
status text not null check (status in ('pending', 'in_progress', 'completed', 'rejected')) default 'pending'
```

## 4. Migrations

All database changes must be tracked in `supabase/migrations/`.

### Creating a Migration
Run `supabase migration new <name>`.
This creates a file like `20231025120000_<name>.sql` (in Ascendra, we manually number them for clarity, e.g., `061_add_task_priority.sql`).

### Immutability
- Never edit a migration file once it has been applied to staging or production.
- If you made a mistake, create a new migration to fix it (e.g., `alter table`, `create or replace function`).

## 5. Triggers

Triggers are used for side-effects that must happen immediately and transactionally within the database.

### The `updated_at` Trigger
Every table must use this trigger to automatically manage the `updated_at` column.

```sql
create trigger set_tasks_updated_at
  before update on public.tasks
  for each row
  execute function public.set_updated_at();
```

# ADR 001: Multi-Tenant Data Isolation with PostgreSQL RLS

## Status
Accepted

## Context
Ascendra is a multi-tenant MLM Leadership SaaS platform serving multiple MLM organizations (Companies). A core security requirement is that users (Leaders and Members) in one company must not be able to read, update, create, or delete data belonging to another company. 

If isolation is managed entirely in the application layer (e.g. NestJS or Flutter clients including `WHERE company_id = ...` in queries), a single developer oversight or query bug could expose one tenant's sensitive information to another. We need a centralized, fail-safe isolation mechanism.

## Decision
We decided to enforce tenant isolation at the database layer using PostgreSQL **Row-Level Security (RLS)** on all tables accessible by client sessions. 

1. **Table Configuration**: Every client-accessible table includes a `company_id` column.
2. **Context Resolution**: The database resolves the current user's tenant by matching `auth.uid()` (the JWT user ID provided by Supabase Auth) against the `profiles` table:
   ```sql
   create or replace function public.get_user_company_id()
   returns uuid language sql stable security definer as $$
     select company_id from public.profiles where auth_user_id = auth.uid() limit 1;
   $$;
   ```
3. **Policy Rules**: Apply a standard policy to all multi-tenant tables:
   ```sql
   create policy "Users can select own company data" 
     on public.tasks 
     for select 
     using (company_id = public.get_user_company_id());
   ```

## Consequences

### Positive Impacts
- **Centralized Enforcement**: Security is configured once in the database schema, protecting data even if application code or external integrations are compromised.
- **Simplified Client Queries**: The database automatically filters results, allowing the Flutter client to issue simple queries (e.g., `select * from tasks`) without manually appending tenant filters.
- **Bypass Protection**: Database-level constraints prevent any inserts or updates that violate tenant boundaries.

### Trade-offs & Cons
- **Connection Overhead**: Resolving `get_user_company_id()` requires querying the `profiles` table. Although indexed, it adds a lookup operation to each query.
- **Service Layer Bypass**: Services that run outside a client session (such as background cron workers in NestJS) must bypass RLS using the Supabase `service_role` key, shifting responsibility back to the application layer for those operations.
- **Testing Complexity**: Writing test suites requires generating authenticated sessions for different users to verify that RLS policies block unauthorized access.
